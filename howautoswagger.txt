\f2\b\fs48 \cf0 \strokec2 Swagger/OpenAPI Discovery Engine \'96 Developer Guide\
\pard\pardeftab720\sa298\partightenfactor0

\fs36 \cf0 Overview\
\pard\pardeftab720\sa240\partightenfactor0

\f0\b0\fs24 \cf0 The purpose of the Swagger Discovery Engine is to automatically locate publicly exposed Swagger or OpenAPI specifications on a target application. Instead of assuming that an API specification exists at a fixed location, the engine performs multiple discovery techniques in a specific order, gradually increasing the search depth while minimizing unnecessary HTTP requests.\
The discovery process is divided into multiple phases. Each phase attempts to locate the API specification using increasingly aggressive techniques. As soon as a valid specification is found, the remaining discovery phases are skipped to reduce network traffic and improve performance.\
A specification is considered valid only after it has been downloaded, parsed successfully, and verified to contain the required OpenAPI or Swagger schema fields.\
\pard\pardeftab720\partightenfactor0
\cf3 \strokec3 \
\pard\pardeftab720\sa321\partightenfactor0

\f2\b\fs48 \cf0 \strokec2 Phase 10 \'96 Authentication Analysis\
\pard\pardeftab720\sa240\partightenfactor0

\f0\b0\fs24 \cf0 The engine inspects the specification for authentication mechanisms.\
Depending on the OpenAPI version, authentication information is extracted from either 
\f3\fs26 securityDefinitions
\f0\fs24  (Swagger 2.0) or 
\f3\fs26 components.securitySchemes
\f0\fs24  (OpenAPI 3.x).\
Supported authentication types include Bearer tokens, API keys, Basic Authentication, Cookie Authentication, and OAuth2.\
Rather than authenticating automatically, the engine records the expected authentication requirements for each endpoint. This information is later used by the security testing engine to compare expected behaviour against actual responses, enabling the detection of broken authentication or authorization issues.\
